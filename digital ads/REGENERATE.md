# Ad Asset Pipeline — How to Regenerate

Brian will ask for a full re-run after final UI/art designs land. The pipeline is fully
scripted; the output adapts automatically to whatever the app looks like at run time
(footage and stills are captured live from prod). Follow this doc top to bottom —
every gotcha below was hit for real the first time.

## Pipeline overview (3 scripts in `_build/`, run in order)

```
node _build/capture.js        # 1. play a real bot match on prod, record video + stills
node _build/render.js         # 2. compose statics/logos/end cards from HTML templates
bash _build/build_videos.sh   # 3. cut + brand the three 15s video formats
```

Prereqs: `npm install` inside `_build/` (playwright), `npx playwright install chromium`,
`brew install ffmpeg` (NOTE: homebrew ffmpeg has NO drawtext filter — that's why the
banner is a PNG overlay, don't try drawtext again).

## Step 1 — capture.js (live footage from prod)

- Creates a throwaway guest via `POST /api/player/guest` (id `guest_chrome_ad*`,
  display name **AceMonkey** — change if Brian wants a different on-screen name).
- Injects `localStorage.poker_user = {playerId, name}` via `addInitScript` → auto-login.
  The init script must guard `if (!document.body) return` and use a plain `setInterval`
  (MutationObserver on documentElement at init time throws and silently kills the script).
- A 250ms interval strips dev chrome from every frame: version watermark (`v5.xxx`)
  and the Feedback pill. **If the UI redesign renames/moves these, update the regexes.**
- Records 720x1280 @ dsf2 (stills land at 1440x2560), starts a bot match, then plays
  ~85s: call/check mostly, raise every 4th action.
- v5.158 lobby: there is NO "PLAY BOT" button anymore. **Do not click QUICK MATCH** —
  it broadcasts a challenge to every human online. The script instead clicks the VS
  button on the house-bot row ("Always ready 🤖"), which starts a bot match instantly;
  that row renders last, so `getByText(/^VS$/).last()` is the bot.
- Facing a shove (`Fold`/`All In` only): the loop now holds 3.5s for drama, then folds
  before the 55s mark and calls all-in after it — ends the run on a showdown +
  match-over modal instead of the old 30s timer stall.
- Selectors are text-based (`Always ready`, `^VS$`, `^Fold$`, `^(Check|Call)`,
  `^(Bet|Raise) [\d,]+$`, `^All In$`, `Play Again`) — **re-check against the UI before running.**

## Step 2 — render.js (statics, logos, end cards)

- Renders HTML templates in Chromium at exact ad pixel sizes and screenshots them.
- MUST write HTML to a temp file and `page.goto(file://...)` — `setContent` makes the
  page `about:blank`, which blocks `file://` images (first run produced empty frames).
- Brand inputs to update for the new art:
  - `FLAG` const → `client/assets/flag-logo.png` (955x626 pirate flag wordmark, the
    brand mark since v5.158; cigar-free). `MONKEY` → `login-monkey.png` (pirate
    portrait, backs the 9:16 end card only — landscape/square use the suits bg
    because the face fights the CTA). Flag is ~3:2: size by WIDTH, never crop round.
  - Colors in `BASE_CSS` → pulled by hand from `client/src/theme.js`
    (navy `#0a1322`, gold `#f0c040`/`#d4a017`). Re-sync with the new palette.
  - Hero screenshots → `SHOT_ALLIN` / `SHOT_SHOWDOWN` / `SHOT_ACTION` consts; after each
    new capture, eyeball `_build/raw/shots/` and repoint to the most dramatic frames
    (all-in moments, showdowns, betting controls visible).
- `.phonebox` needs `overflow:hidden` (screenshot bleeds past the fade otherwise);
  landscape-logo wordmark needs `white-space:nowrap` or it wraps.
- Headlines/copy variants live inline in the `jobs` array at the bottom.

## Step 3 — build_videos.sh (three 15s MP4s)

- Cuts TWO segments from `raw/gameplay.webm` and fades into a per-aspect end card:
  segment timestamps are HARDCODED (2026-06-11 capture: `-ss 46 -t 6.5` + `-ss 79.5 -t 6`) and MUST be
  re-picked for every new capture. To pick: extract probe frames —
  `ffmpeg -ss <t> -i raw/gameplay.webm -frames:v 1 -vf scale=270:-1 probe/f<t>.png`
  every ~6s, view them, choose (a) a window with betting controls visible and
  (b) an all-in/showdown window. Total gameplay 12.5s, end card 3s, xfade 0.4s
  at offset 12.1 → exactly 15.0s.
- The top dev strip is covered by `_endcards/banner_720x50.png` (regenerated by
  `_build/_banner.js` — update its text/colors for the rebrand). PNG overlay, not
  drawtext (missing in homebrew ffmpeg), and beware: a literal apostrophe in any
  filtergraph string breaks parsing — use the typographic ’ character.
- 1:1 and 16:9 use blurred-self pillarbox; 9:16 is a direct lanczos upscale.
- Silent by design + `anullsrc` AAC track (some platforms reject zero audio streams).
  Brian adds music later: `ffmpeg -i vid.mp4 -i track.mp3 -map 0:v -map 1:a -c:v copy -shortest out.mp4`.

## Step 3b — animate.js (motion-graphics ad, the "marketing-style" video)

`node _build/animate.js` → `video/poker_monkey_15s_9x16_motion.mp4` (9:16, 15s, 30fps).

- Scene timeline, all timings in ms inside the script: logo hook (0–2.2s) → headline
  card "All in. / Zero cost." (2.2–4.6s) → floating tilted phone playing real gameplay
  (4.6–11.8s) → CTA end card with pulse (11.8–15s). Suit-pattern background drifts
  throughout.
- How it works: one HTML page, every animation built with element.animate() and
  immediately paused; `window.seekTo(ms)` scrubs all animations AND the embedded
  <video> to an exact timestamp; node loops 450 frames at 33.3ms steps, screenshots
  each, ffmpeg assembles. Fully deterministic — re-runs are pixel-identical.
  ~2 min render.
- Needs `raw/ad_clip.webm` first (built in this doc's Step 3 prep; 2026-06-11 capture
  used `-ss 47 -t 3.5` + `-ss 83 -t 3.7`):
  `ffmpeg -ss 60 -t 3.5 -i raw/gameplay.webm -ss 91.8 -t 3.7 -i raw/gameplay.webm \
   -filter_complex "[0:v][1:v]concat=n=2:v=1,fps=30[v]" -map "[v]" -c:v libvpx-vp9 -crf 24 -b:v 0 raw/ad_clip.webm`
  (re-pick those timestamps per capture). MUST be VP8/VP9 webm — Playwright's
  Chromium has no H.264 decoder, an mp4 will render a black phone screen.
- Video seeking: each seek awaits the 'seeked' event with a 400ms safety timeout.
  Without the timeout the frame loop hangs on missed events.
- For 1:1 / 16:9 motion versions: change W/H consts and re-flow the CSS (font sizes
  and the phone width are absolute px tuned for 1080x1920).
- The phone is a full CSS phone body (titanium frame, bezel, camera island, side buttons, glass shine) — island intentionally covers the gameplay header strip.
- Rebrand inputs: same as render.js — LOGO const, colors in the <style> block,
  headline/CTA copy in the HTML body.

## Step 4 — package & ship

- Curate ~6 stills from `_build/raw/shots/` into `screenshots/` with semantic names.
- Deliverables checklist (what Google/Meta asked for):
  videos 16:9 + 1:1 + 9:16 · statics 2x 4:5 (1080x1350) + 1:1 (1080x1080) +
  1200x628 · logos 1200x1200 + 512 + 4:1 · headlines/descriptions/primary text
  in `copy.md` (refresh char counts if reworded) · landing URL.
- `_build/raw/` + `_build/node_modules/` are gitignored (heavy, regenerable).
- Multi-session warning: last run, a concurrent session switched the checkout's branch
  mid-task and my commit landed on their feature branch — check `git branch --show-current`
  immediately before committing, and if needed push the commit SHA directly:
  `git push origin <sha>:main` (works as fast-forward).
- Optional cleanup: delete the `guest_chrome_ad*` player from prod afterwards
  (id saved in `_build/raw/player.txt`).

## Policy notes that carry over regardless of art

- Keep "Free to play · No real money" on every creative + in copy (social poker, not gambling).
- Current logo's cigar risks Meta tobacco-imagery disapproval — if the new art keeps it,
  have a cigar-free crop ready as a fallback.
